# ADR-0014: Operator Web 保持服务端多页面模板壳，不升级为独立 SPA

## 状态

已采纳

## 背景

`proxy-platform` 的 operator Web 已经正式部署上线，但最初实现只是为了验证远端 operator 工作台闭环，因此页面长期停留在 `web_app.py` 内联 HTML/CSS/JS 的形态。

这带来三个问题：

1. 页面在人机交互上更像“工程验证页”，不适合长期值守。
2. 视图、样式、脚本和后端路由写在同一个文件里，后续扩展风险高。
3. 如果直接借机升级为独立 SPA，又会把当前薄平台壳推向新的重前端维护面。

## 决策

operator Web 继续保持：

- `FastAPI` 负责 API、认证和页面壳层；
- 页面改为服务端模板 + 本地静态资源；
- 页面信息架构改成共享壳层 + 多个职责页，而不是继续堆成一个大页面；
- 交互只保留少量原生 JS；
- 不引入独立前后端分离工程，不把 `apps/web-console` 立刻升级为正式前端主入口。

## 设计含义

在这个项目里，这个决策的意思不是“反对前端工程化”，而是：

- 当前站点本质上还是 operator 工作台，不是产品级控制面；
- 当前更重要的是把“总览、主机、订阅、provider、作业、审计”按 operator 任务拆开；
- 页面层的复杂度应该被限制在薄壳范围内，不能盖过平台合同本身。

## 为什么这样设计

### 它解决什么

- 把页面从 `web_app.py` 的大段内联字符串里拆出来，后续可维护。
- 给 operator 一个更清晰的现场工作台，而不是继续看调试页。
- 保持单容器、易部署、易回滚，不额外增加前端构建链和静态站点发布链。

### 它不解决什么

- 不直接提供新的 northbound 控制面。
- 不提供实时 agent 通信或复杂前端状态管理。
- 不替代未来真正需要时才可能建设的完整 `apps/web-console`。

## 后果

### 正面后果

- 页面结构、样式和脚本的职责边界更清楚。
- operator Web 可以更自然地继续扩展摘要卡、搜索、toast、审计卡片等能力。
- operator 不需要在一个长滚动页里同时处理“看现场”和“做变更”。
- 远端部署仍然保持单容器路径，不引入第二套发布复杂度。

### 负面后果

- 当前页面仍然不是完整 SPA，复杂交互不会像独立前端那样自由。
- 如果未来页面复杂度持续快速增长，需要重新评估是否升级为正式前端工程。

## 与其他方案的比较

### 方案 A：继续在 `web_app.py` 内联扩展

不采纳。原因是 UI 复杂度和 API 路由会继续缠在一起。

### 方案 B：立刻重建独立 SPA

不采纳。原因是当前仓库边界和产品阶段都不支持这样做，维护成本明显高于收益。

### 方案 C：服务端模板壳 + 本地静态资源

采纳。这是当前阶段最符合薄平台壳定位、又能明显改善 operator 可用性的方案。
